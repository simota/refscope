### Tomo: 数万 commit の OSS メンテナー

## Request: 1 ファイルがどう育ったかを、timeline 全体に焦点を合わせずに辿りたい

**Speaker:** Tomo (パワーユーザー x OSS メンテナー x 数万 commit の monorepo)
**Scene:** 古いバグレポを再現中、`src/lib/parser.ts` の 2 年分の変遷を把握したい。path filter で絞り込むが、200 件で止まる。

### User Voice (First Person)
> path filter は便利ですが、200 件で止められると過去への旅が砂壁にぶつかります。私は `src/lib/parser.ts` の歴史を、年単位で深く掘りたい。timeline 全体の見た目を変えずに、このファイルだけの page-through と year-jump をください。それと、リネームを越えてもらいたい — 名前が変わった瞬間に history が途切れたら、私は嘘の context で判断することになります。

### Why This Is Needed
- 大規模 repo では 200 commit window は数日分にもならず、長期メンテナーの調査用途を満たさない。
- リネームを跨げない history は途切れた事実を見せ、誤った判断の原因になる。
- file-level の履歴は repo 全体の overview とは別の作業なので、UI で混ぜずに分けて欲しい。

### Acceptance Criteria (User Perspective)
- [ ] path filter 適用時、過去方向へ cursor ベースで継続ページングでき、200 件超でも調査が続けられる。
- [ ] リネーム検出が有効化され、リネーム前後の history が繋がって見える。リネームの根拠 (検出方法) も明示される。
- [ ] file-level の history view が timeline と分離されており、対象ファイルの commit metadata と該当ファイルのみの diff に集中できる。

### Emotional Impact
- **Current emotion:** 焦り

### Yuki: ADHD / sensory-sensitive な開発者

## Request: 色とアニメーションを止め、必要なときだけ live updates を許す静かなモードがほしい

**Speaker:** Yuki (日常ユーザー x ADHD / sensory-sensitive x キーボード + 集中作業)
**Scene:** diff を集中して読んでいる最中、top bar の live pulse が動き、sidebar の bar chart が更新されて視線が逸れる。

### User Voice (First Person)
> visual summary は素敵だし役に立つときもあります。でも私が code を読んでいる時、それは敵です。色を抑えて、アニメーションを止めて、live updates も明示的に許可した時だけ走らせる『静かなモード』をください。pause だけでは足りません — 読んでいる最中に状態がチラチラするのを止めたい。`prefers-reduced-motion` を尊重してくれるなら、それだけで救われる人もいます。

### Why This Is Needed
- sensory sensitivity / ADHD のユーザーは色とモーションで集中を失いやすい。
- 既存の live updates pause は SSE 反映を止めるが、charts や hover、色彩強調までは止まらない。
- 読みやすさはアクセシビリティの問題であり、同時に集中作業の生産性の問題でもある。

### Acceptance Criteria (User Perspective)
- [ ] 「Quiet mode」を top bar / command palette / OS の `prefers-reduced-motion` 設定から有効化できる。
- [ ] quiet mode 中: 補助的アニメーション (pulse、chart の更新トランジション) が停止し、彩度・コントラスト強調が抑えられ、live updates が pause になる。
- [ ] quiet mode の出入りで、現在の repo / ref / commit selection、scroll 位置、active filter が失われない。
