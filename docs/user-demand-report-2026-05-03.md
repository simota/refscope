## Request: PR を出す前に「見落としている未完了作業」を平易に教えてほしい

**Speaker:** Nao (週 2 日の junior contributor × Git 初心者 × PR 前のセルフチェック)  
**Scene:** 初めて大きめの修正を終え、PR を出す直前。timeline には commit、上には working tree pseudo-row、sidebar には stash / worktree / submodule が出ているが、Nao はそれぞれの意味を正確に知らない。

### Acceptance Criteria (User Perspective)

- [ ] 「これは Refscope の推測です」と「これは Git から見えた事実です」が分かれている。
- [ ] チェック結果を PR description に貼れる短い Markdown としてコピーできる。

- **Current emotion:** 不安
- **Post-fulfillment emotion:** 安心

### Priya: release captain

### Mateo: review-heavy senior engineer

## Request: レビューすべき順番を、関連ファイルと変更の広がりから組み立てたい

**Speaker:** Mateo (review-heavy senior engineer × 大規模 monorepo × CI 待ちの並行レビュー)  
**Scene:** 夕方、3 本の PR と 1 本の hotfix を同時に見ている。Refscope の file history と related files は便利だが、どの commit から読み始めればよいかはまだ自分で考えている。

### User Voice (First Person)
> related files が出るのは助かります。でも私が知りたいのは「この変更はどこまで広がっているか」と「レビューはどこから読むと一番早く危険に当たれるか」です。commit を新しい順に読むだけだと、最後に migration や shared utility の変更に気づくことがあります。変更ファイル、関連ファイル、diff の大きさ、テストファイルの有無から、レビューの読み順を提案してほしい。ただし、これは推測だと分かるようにして、根拠のファイル一覧を必ず見せてください。

### Why This Is Needed

- 大規模 repo のレビューでは、時系列順が最適な読み順とは限らない。
- related files は観測された co-change だが、「どこから読むべきか」は派生判断なので、根拠付きで提示する必要がある。
- レビュー担当者は自動判断を盲信したいのではなく、危険に早く到達するための入口を欲しがっている。

### Acceptance Criteria (User Perspective)

- [ ] 選択 commit / worktree changes に対し、review order suggestion が表示される。
- [ ] suggestion は `derived` と明示され、根拠として changed files、related files、diff size、test file presence が見える。
- [ ] 「先に読む」「後で読む」「確認不要かも」のような行動語で grouping される。
- [ ] suggestion を無効化でき、通常の timeline / diff 体験を邪魔しない。

### Emotional Impact

- **Current emotion:** 苛立ち
- **Post-fulfillment emotion:** 集中
- **User-felt urgency:** 日常的なレビュー負荷

## Questions for the Team

1. Refscope は観測ツールに留まるのか、それとも `derived` ラベル付きで「次の行動」を提案するのか。
2. stash / worktree / submodule / operation を別々の sidebar section として見せるだけで、PR 前・release 前のユーザー判断は本当に支援できているのか。
3. live updates は全画面に常に流すべきか、それとも release check / PR-ready check のような判断画面では snapshot 時刻を固定すべきか。
4. `derived` な suggestion を出す場合、誤判定時の責任を UI 文言と根拠表示でどう限定するのか。

## Don't Build Option

- PR-ready / release-check / review-order を一気に実装しない。まずは既存観測 API を使った read-only checklist だけで、ユーザーが「判断が速くなった」と感じるか検証する。
- 自動修正、stash apply、worktree delete、submodule update などの破壊的・状態変更操作は入れない。今回の需要は「判断」支援であり「Git 操作」支援ではない。
