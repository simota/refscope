### Lin: deuteranomaly のフロントエンドエンジニア

## Request: syntax highlight の色が diff の add/del 色と区別できないので、色以外の signal がほしい

**Speaker:** Lin (日常ユーザー × フロントエンドエンジニア × 色覚多様性 (deuteranomaly / 第二色弱))
**Scene:** 今朝、Refscope の最新版に更新して diff viewer で TypeScript の関数を読んでいる。Prism syntax highlighting の keyword 色 (緑系) と「追加行」の緑が、彼にはほぼ同じトーンに見える。「`function` キーワードが新しく増えた行に書いてあるのか、もとからあったのか」が判別できず、いちいち先頭の `+/-` 記号と左バーで確認している。

### User Voice (First Person)
> syntax highlighting を入れてくれたのは嬉しいです。でも、私には keyword 色と「追加行の緑」が同じに見えます。色だけが情報のレイヤーになっていると、私は色覚補正眼鏡を使うか、`+/-` だけを頼りに読むかしかありません。色は飾り、情報は文字とアイコンと位置で先に伝えてほしい。Prism の theme を切り替えるか、status を `A`/`M`/`D` のテキストや左バーの太さで伝えるかしてくれませんか。Quiet mode は彩度を落としてくれるけれど、hue (色相) が衝突している私の問題は彩度では解けません。

### Why This Is Needed
- color-blind の頻度は男性で 5-8% (deuteranomaly が最多)。dev tool であるほど無視できない母集団。
- 既存の Quiet mode は彩度方向の介入で、syntax highlight の hue 衝突は彩度では解決しない。orthogonal な軸が必要。
- diff の add / del は Git status (`A`/`M`/`D`) という観測事実なので、テキストやアイコン (色以外のチャネル) でも表現できる。色だけの情報チャネルには冗長性が必要 (WCAG 2.2 1.4.1 Use of Color)。
- 開発者が dev tool で色だけに頼ると、自分が書く web app でも色だけに頼る悪循環を作る。

### Acceptance Criteria (User Perspective)
- [ ] diff の各行に色以外の signal (左のバーの幅・太さ、`+`/`-` 字、文字色のコントラスト差) があり、grayscale でも add / del / context が読める。
- [ ] Prism theme を最低 2 種類 (現行と high-contrast / color-blind safe) から選べる。OS の `prefers-color-scheme` も尊重する。
- [ ] color-blind safe theme は deuteranomaly / protanomaly で keyword と diff add 色が判別可能 (グレースケール変換でもコントラスト 3:1 以上)。
- [ ] Quiet mode との独立性が保たれる (Quiet で彩度が落ちても、color-blind safe theme と組み合わせ可能)。
- [ ] WCAG 2.2 AA の non-text contrast (3:1) を満たす。

### Emotional Impact
- **Current emotion:** 諦め (毎回個別対応)
- **Post-fulfillment emotion:** 信頼 (この tool は私を考えている)
- **User-felt urgency:** 日常 (diff を読むたびに発生)

### Riku: pickaxe を日常使う SRE

## Request: diff content (pickaxe) と grep で commit を探したい — subject 検索だけでは届かない

**Speaker:** Riku (パワーユーザー × SRE × CLI fluent)
**Scene:** 先週、production のあるロギング処理が知らぬ間に削られていた疑いがあり、`git log -S "logger.warn"` で消失タイミングを追っていた。Refscope の検索ボックスに `logger.warn` を入れたが commit subject にしか hit せず、結局 CLI に戻って `git log -S` を叩いた。Refscope の存在意義が「CLI を補完する viewer」なのに、肝心の調査では使えない。

### User Voice (First Person)
> 私の調査の 7 割は subject じゃなくて diff の中身を探すことです。`-S "OAUTH_SECRET"` で消失 commit を当てる、`-G regex` で正規表現にマッチする差分行を探す、`--grep` で commit message を正規表現で絞る、これは Git native の話で、皆さんが UI 側で発明する必要はないんです。検索ボックスをモード切替にしてくれませんか。subject / pickaxe / grep / path / author で。それぞれ Git の挙動に合わせれば、私は CLI に戻らなくて済む。Refscope 独自の検索 DSL を作られると、私は学習コストが二重になって、結局 CLI を選びます。

### Why This Is Needed
- Refscope は CLI 補完を志向するが、現在の filter は subject + author + path + date のみ。実調査用途の中核 (pickaxe `-S` / `-G`、message regex `--grep`) が欠落。
- power user は filters の裏に Git native 機能があると期待する。「Refscope 独自の検索 DSL」を作ると学習コストが二重化して逆効果。
- pickaxe (`-S` / `-G`) は Git の事実。Refscope が prediction で結果を変えてはいけない (round 2 journal の「観測 vs 派生」原則と整合)。
- incident / code archeology の use case は週次〜日次で発生する。CLI に戻るたびに Refscope への信頼が下がる。

### Acceptance Criteria (User Perspective)
- [ ] 検索モード切替 (`subject` | `pickaxe -S` | `regex -G` | `message --grep` | `path` | `author`) が UI で明示的に選べる。
- [ ] 各モードが対応する Git option を tooltip / help text で開示する (ユーザーが裏で何が走っているか分かる)。
- [ ] 検索結果に一致根拠 (どの行 / どの commit / どの ref) が表示され、なぜ hit したかが追える。
- [ ] 範囲制限 (date range, ref) が現状の filter UI と互換し、組み合わせで効く。
- [ ] 大きい repo で重いことが事前に分かる (estimated cost / streaming / cancel)。

### Emotional Impact
- **Current emotion:** 苛立ち (CLI に戻る毎回の摩擦)
- **Post-fulfillment emotion:** Refscope を「調査の primary tool」と認める
- **User-felt urgency:** 週次〜日次 (incident や code archeology のたび)

### Asha: 多顧客の業務委託エンジニア

## Request: screen share / screenshot の前に、author email など個人情報をワンタッチで隠したい

**Speaker:** Asha (共有環境ユーザー × 業務委託エンジニア × 複数クライアント repo の screen share 担当)
**Scene:** 今日の振り返り MTG で、commit author の `tanaka@client-corp.co.jp` が画面共有中に映ったことに後で気づいた。screenshot を Slack に貼った後にも気づいて慌てて消したが、ログには残る。彼女は 1 日 5-6 回 Zoom で repo を share する。次の打ち合わせまでに、毎回 dev tools で要素を消すのは現実的でない。

### User Voice (First Person)
> 私は仕事柄、複数のクライアントの repo を screen share します。Refscope はとても見やすいから使いたい。でも、author の email、author name のフルネーム、reviewer の email、SHA に紐づく Slack username なんかが、私が screen share する 5 秒の間に、第三者の目に入ってしまいます。設定画面に深く潜らず、トップバーに「Redact」みたいな 1 ボタンが欲しい。on にすると author email を `***` に、author name を頭文字に、長い hash を短縮に、commit message の `Reviewed-by:` 行を伏せる、みたいな。off に戻したら元に戻る。Git の中身は変えなくていい、私の画面だけ守ってくれればいい。あと、隠しているつもりだったが実は出ていた、という事故が一番怖いので、redaction が ON であることを目立つ場所で叫んでください。

### Why This Is Needed
- 業務委託 / 多顧客環境では情報漏洩は契約違反の可能性。
- 多くの dev tool は「自分の画面に映って当たり前」と考えるが、screen share / screenshot 前提で生きるユーザーは少なくない (contractor、教育、登壇、bug report、support ticket)。
